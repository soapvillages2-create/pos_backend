# POST `/api/qr/sync-menu` — ค่า `shopId` กับ JWT (แก้ HTTP 403)

## นิยามทางเทคนิค

| ค่า | ความหมาย |
|-----|-----------|
| **`shopId` ใน body** | ต้องเป็น **string เดียวกับ `tenants.tenant_id`** ของร้านนั้น (รูปแบบเช่น `shop` + เลข 6 หลัก ตามที่ระบบสร้างตอนสมัคร) |
| **JWT `tenantId` claim** | มาจาก `users.tenant_id` ตอน login — **ต้องเท่ากับ `shopId` แบบเปรียบเทียบ string หลัง trim** |
| **ไม่มี mapping อื่น** | ไม่มี `storeId` / UUID แยก — ใช้ **`tenant_id` เป็นตัวเดียว** |

โค้ดตรวจ: `requireShopMatch` ใน `controllers/qrController.js` (`shopId === req.user.tenantId` หลัง `String` + `trim`).

## เหตุผล HTTP 403 (`ไม่ตรงกับร้านที่ล็อกอิน` / `SHOP_TENANT_MISMATCH`)

**ออกแบบมาเพื่อกันสลับร้าน:** ผู้ใช้ล็อกอินด้วย user ของร้าน A แต่ส่ง `shopId` ของร้าน B (หรือพิมพ์ผิด / คัดลอกฟิลด์ผิด) จะได้ **403** — ไม่ให้เขียนเมนูข้าม tenant.

กรณีอื่น:

- **`shopId` ว่าง** → **400** + `code: SHOP_ID_REQUIRED` (เฉพาะ `sync-menu` หลังอัปเดต)
- **`shopId` ตรง JWT แต่ไม่มีแถวใน `tenants`** → **404** `ไม่พบร้านค้า` (เคส DB ผิดปกติ)

## คำแนะนำ 1 ย่อหน้า (ให้ผู้ใช้ / ซัพพอร์ต)

**ให้ตั้งค่า Shop ID ในแอป POS ให้ตรงกับรหัสร้าน (Tenant ID) ของบัญชีที่ล็อกอิน — ค่านี้คือ `tenant_id` ในระบบหลังบ้าน ไม่ใช่ชื่อร้านหรือเลขอื่นที่คล้ายกัน** หลังล็อกอินสำเร็จ API ส่ง `tenantId` ใน response (เช่น `data.user.tenantId` หรือฟิลด์ที่ทีมแอป map ไว้) — ใช้ค่านั้นเป็นต้นฉบับ หรือดูจากหน้า/รายงาน Admin ที่แสดง **รหัสร้าน / Tenant ID** ของร้านนั้น แล้วคัดลอกมาใส่ในตั้งค่า POS ให้ตรงทุกตัวอักษร (รวมคำว่า `shop` ถ้ามี) จากนั้นกด Sync เมนูอีกครั้ง

*(ถ้า Admin ยังไม่มีหน้าแสดง `tenant_id` ชัดเจน — แนะนำให้ทีมผลิตภัณฑ์เพิ่มฟิลด์ “รหัสร้าน (Tenant ID)” ในหน้าตั้งค่าร้าน หรือให้ user ดูจาก response หลัง login / หน้าโปรไฟล์ในแอป)*

## คำตอบสั้น ๆ ถึงทีมแอป

1. **ค่า `shopId` ที่ถูกต้อง** = **`tenant_id`** ของผู้ใช้ที่ล็อกอิน (เดียวกับ claim **`tenantId`** ใน JWT).
2. **ต้องเท่ากันแบบ string ตรง ๆ** (หลัง trim) — ไม่มีแปลงรูปแบบอื่น
